# Bug Fixes - June 14, 2026

## 🐛 BUG 1: Invite Link Password Setup Fails with "Auth session missing!"

### Problem
When invited users clicked their setup link and tried to set a password, they got "Auth session missing!" error. The password update would fail because no session was established.

### Root Cause
The `set-password.html` page only checked for **hash token format** (`#access_token=...&refresh_token=...`) but modern Supabase invite links use **PKCE flow** (`?code=...`). The page never called `exchangeCodeForSession()`, so no session was established before `updateUser({ password })` was called.

### Fix Applied

#### 1. Updated `/app/set-password.html`

**Before**: Only checked URL hash for tokens
```javascript
const hashParams = new URLSearchParams(window.location.hash.substring(1));
const accessToken = hashParams.get('access_token');
```

**After**: Handles both PKCE and hash token flows
```javascript
// Check for PKCE code in query params
const pkceCode = urlParams.get('code');

if (pkceCode) {
  // Exchange PKCE code for session
  const { data, error } = await sb.auth.exchangeCodeForSession(pkceCode);
  session = data.session;
} else if (accessToken && refreshToken) {
  // Fallback: manual session from hash tokens
  const { data, error } = await sb.auth.setSession({ access_token, refresh_token });
  session = data.session;
}
```

#### 2. Added Session Verification Before Password Update
```javascript
// Verify session exists before updating password
const { data: { session } } = await sb.auth.getSession();
if (!session) {
  throw new Error('Auth session missing! Please click the invitation link again.');
}
```

#### 3. Improved Error Messages
- ✅ "This invitation link has expired or was already used" - for PKCE code reuse
- ✅ "Invalid or expired invitation link" - for missing tokens
- ✅ Clear distinction between PKCE vs hash token errors

#### 4. Role-Based Routing After Password Set
After password is successfully set, users are automatically routed to their workspace:
- `super_duper_admin` → Master Console
- `super_admin` → Company Dashboard
- `admin/admin_m/sdr/account_executive` → CRM (if enabled) or Sales OS

#### 5. Email Scanner Mitigation (CRITICAL)

**Problem**: Email security scanners (Gmail, Outlook, Microsoft Defender) pre-fetch links to check for malware. This consumes single-use tokens BEFORE the real user clicks, causing instant "expired" errors.

**Solution - Two-Stage Activation**:

**Stage 1 - Pre-activation (Safe for scanners)**:
```javascript
async function init() {
  // Check if tokens EXIST but DO NOT EXCHANGE yet
  const pkceCode = urlParams.get('code');
  
  if (!pkceCode) {
    showExpiredError();
    return;
  }
  
  // Store token info, show button - NO API calls
  inviteTokens = { type: 'pkce', pkceCode };
  showPreActivationButton(); // "Continue to Set Password"
}
```

**Stage 2 - Activation (User action required)**:
```javascript
async function activateInvite() {
  // Called ONLY when user clicks "Continue to Set Password"
  // This ensures scanners don't consume token on background fetch
  
  const { data, error } = await sb.auth.exchangeCodeForSession(inviteTokens.pkceCode);
  // Token consumed only here - after explicit user interaction
}
```

**Why this works**:
- Email scanners perform silent GETs without JavaScript or user interaction
- They see HTML page but never trigger button click
- Token remains unconsumed until real user clicks
- PKCE code (`?code=`) is better than hash tokens (`#access_token=`) because scanners fetch the URL

#### 6. Extended Link Lifetime

**Recommendation**: Configure Supabase to increase email link expiry:
- **Default**: 60 seconds (too short!)
- **Recommended**: 3600 seconds (1 hour) or 7200 seconds (2 hours)

**Configuration**: Supabase Dashboard → Authentication → Email → Email Link Expiry = `3600`

**Why**: 
- Users may not click immediately after receiving email
- Admins copy-paste links manually (no email sent)
- Reduces pressure on single-use token window

#### 7. Clear Error Recovery

Added dedicated expired link UI:
```html
<div id="expired-link">
  ⏱️ Link Expired or Already Used
  
  This invitation link has expired or was already used.
  
  [ Request New Invitation Link ]
  
  Contact your administrator if you need assistance.
</div>
```

Clicking "Request New Invitation Link" shows instructions to contact admin.

## 🐛 BUG 2: Inviting Existing User Shows Cryptic "such an email doesn't exist" Error

### Problem
When reenusha19 (super_admin) tried to invite shaamel1970@gmail.com, the invite flow failed with error "such an email doesn't exist" - but that email ALREADY existed as an admin_m user. The error message was misleading and didn't explain the real issue.

### Root Cause
The `generate-invite-link` Edge Function tried to create a NEW auth user with `auth.admin.createUser()` without first checking if the email already exists. When the email existed, Supabase returned a cryptic error that was passed directly to the UI.

### Fix Applied

#### Updated `/supabase/functions/generate-invite-link/index.ts`

**Added duplicate email check BEFORE attempting user creation:**

```typescript
// Check if user already exists in auth.users
const { data: existingAuthUser } = await supabaseAdmin.auth.admin.getUserByEmail(email)

if (existingAuthUser && existingAuthUser.user) {
  // User already exists in auth - check if they're in the users table
  const { data: existingProfile } = await supabaseAdmin
    .from('users')
    .select('id, email, name, role, org_id, is_active')
    .eq('id', existingAuthUser.user.id)
    .single()

  if (existingProfile) {
    throw new Error(`User with email ${email} already exists in the system. Email addresses must be unique across all organizations. Please use a different email address or contact the existing user's administrator.`)
  }

  // Auth user exists but no profile - orphaned record
  throw new Error(`An account with email ${email} exists but is incomplete. Please contact support to resolve this issue.`)
}
```

**Clear error messages:**
- ✅ "User with email X already exists in the system. Email addresses must be unique across all organizations."
- ✅ Instructs user to use different email or contact existing user's admin
- ✅ Handles orphaned auth records (edge case)

### Alternative Solution (Not Implemented Yet)
Future enhancement: Instead of blocking, offer to **reassign** the existing user:
1. Detect existing user
2. Show confirmation: "User X already exists. Update their role and manager?"
3. If confirmed, update existing user's role/manager/org instead of creating new user
4. Generate new login link for them

This would allow cross-org transfers or role changes via invite flow.
